# Control Unit

Устройство управления формирует управляющие сигналы для DataPath. Работает на
основе микроопераций, которые хранятся в памяти микрокода (MICROCODE MEMORY).
Результат работы CU на каждом такте - набор управляющих сигналов. Физически это
большое двоичное число, где каждый бит соответствует одному сигналу (состав
описан ниже).

## Опкод инструкции

Перед разбором микроопераций стоит вспомнить формат опкода, так как CU
анализирует его поля при переходах внутри микрокода:

```
[lb:1][instr:4][instr_type:3][arg1_type:3][arg2_type:3][is_addr_dest:1][rsv:1]
```

**lb:**
- 1 - long (операнд 4 байта)
- 0 - byte (операнд 1 байт)

**instr_type:**
- 000 - system (hlt, nop, mov)
- 001 - alu
- 010 - control
- 011 - io

**arg1_type / arg2_type** - типы операндов:
- 000 - no op
- 001 - (Ax)
- 010 - d(Ax)
- 011 - d(Ax, Yx)
- 100 - imm
- 101 - регистр
- 110 - -(Ax)
- 111 - (Ax)+

**is_addr_dest:**
- 1 - приёмник = arg1
- 0 - приёмник = arg2

**instr** (значение зависит от instr_type):

```
instr_type == system (000)
  0000 - hlt
  0001 - nop
  0010 - mov

instr_type == alu (001)
  0000 - cmp     1000 - xor
  0001 - add     1001 - asl
  0010 - sub     1010 - asr
  0011 - mul     1011 - lsl
  0100 - div     1100 - lsr
  0101 - not     1101 - pass
  0110 - or      1110 - inc
  0111 - and     1111 - dec

instr_type == control (010)
  0000 - jmp     0101 - bvs
  0001 - beq     0110 - bcs
  0010 - bne     0111 - bcc
  0011 - bmi     1000 - call
  0100 - bpl     1001 - ret

instr_type == io (011)
  0000 - in
  0001 - out
```

## Микрокоманды

Каждая микрокоманда занимает 84 бита. "Микроадрес" - 9 бит. Есть счётчик
микрокоманд µPC, указывающий на адрес следующей исполняемой микрокоманды.

Два типа микрокоманд: simple и branch (различаются по старшему биту).

### Simple

Набор сигналов плюс возможные подстановки.

```
1[mode:4][signals:79]
```

Существует 4-битный регистр `reg`, в который защёлкивается номер регистра из
ИНСТРУКЦИИ (поле op). Вместо явных сигналов для mux1, mux2 или latch'ей регистров
можно подставить значение из `reg` - это упрощает микрокод: одна и та же
микропрограмма работает с любым регистром.

**mode:**
- 0000 - простой набор сигналов
- 0001 - защёлкнуть регистр по `reg:4`
- 0010 - выбрать mux1 по `reg:4`
- 0100 - выбрать mux2 по `reg:4`
- 1000 - выбрать операцию ALU по полю `instr` опкода

### Branch

Переходы внутри микрокода.

```
0[type:15][template:4][addr:9][oth:55]
```

Переход выполняется при совпадении части IR (или NZVC) с шаблоном `template`:
например `0instr_type == template` -> переход, иначе следующая микрокоманда.
Также переход может быть по неготовности io/dm/im (`not_im_rdy` и т.п.),
безусловным по 9-битному адресу, либо по ja1/ja2 (jump address 1/2) для
вложенности подпрограмм микрокода до глубины 2.

**type:**
- `000000000000001` - if 000n == template
- `000000000000010` - if 000z == template
- `000000000000100` - if 000v == template
- `000000000001000` - if 000c == template
- `000000000010000` - if 0arg1_type == template
- `000000000100000` - if 0arg2_type == template
- `000000001000000` - if instr == template
- `000000010000000` - if 0instr_type == template
- `000000100000000` - безусловный jmp
- `000001000000000` - jmp по ja1
- `000010000000000` - jmp по ja2
- `000100000000000` - if not_im_rdy
- `001000000000000` - if not_dm_rdy
- `010000000000000` - if not_io_rdy
- `100000000000000` - if is_addr_dest

## Структура микрокода

Микрокод организован по блокам (метки):

- `instruction_fetch` - выборка опкода, защёлкивание IR, ветвление по
  `instr_type` на одну из четырёх групп (system / alu / control / io).
- Группы (`system`, `alu`, `control`, `io`) - ветвление по полю `instr` на
  обработчик конкретной инструкции.
- Подпрограммы загрузки операндов (`imm1`, `register1`, `imm2`, `register2`,
  `addr_fetch` и режимы `Ax_load`, `dAx_load`, `dAxYx_load`, `mAx_load`,
  `Axp_load`) - вызываются через ja1/ja2, считывают операнд нужной адресации.
- Подпрограммы доступа к памяти (`dm_read`, `dm_write`, `io_read`, `io_write`) -
  содержат циклы ожидания готовности (`not_*_rdy`).
- Обработчики инструкций (`mov_op`, `alu_exec`, `jmp_op`, `beq_op` и др.,
  `call_op`, `ret_op`, `in_op`, `out_op`).

Подпрограммы вынесены отдельно и переиспользуются разными инструкциями, что
заметно сокращает объём микрокода. Глубина вложенности ограничена двумя
уровнями (ja1 + ja2).

## Микрокод (исходник и трансляция)

Писать микрокод напрямую в бинарном виде тяжело, поэтому был придуман 
недо-ассемблер,который транслируется в бинарник, а тот уже 
исполняется моделью.

Синтаксис повторяет форматы микрокоманд:

```
sig [t.<type>] <signal> [<signal> ...]
jmp [t.<type>] [<template>] [<addr>]
```

Поддерживаются метки - вместо них при трансляции подставляется адрес (номер)
микроинструкции.

См. [microcode_generator.py](../processor_model/control_unit/microcode_generator.py)
и [microcode.txt](../processor_model/control_unit/microcode/microcode.txt).
